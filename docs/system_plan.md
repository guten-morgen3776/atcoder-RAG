# AtCoder-RAG システム設計・開発手順書

## 1. システム概要
過去のAtCoderの問題を「曖昧な記憶」や「自然言語のキーワード」から検索し、解法のヒントを提示するRAG（Retrieval-Augmented Generation）システム。
ローカルのPCリソース（ストレージ容量や計算能力）を圧迫しないよう、重い処理（LLM推論やベクトル化）は外部APIにオフロードする構成とする。

### 技術スタック
* **言語:** Python 3.x
* **メタデータ取得:** AtCoder Problems API
* **クローリング:** `requests` + `BeautifulSoup4`
* **テキスト抽出・整形:** Gemini 2.5 Flash API (高速・安価・高精度な要約用)
* **ベクトル化 (Embedding):** OpenAI API (`text-embedding-3-small`) または Gemini API
* **ベクトルDB:** ChromaDB (Python組み込み・ローカルファイル保存)
* **フロントエンド:** Streamlit

---

## 2. データ構造（スキーマ定義）
ベクトルDB（ChromaDB）に格納するデータの構造。検索用ベクトルと、表示・RAG用メタデータを分離する。

| カラム名 | データ型 | 格納する内容 | 役割 |
| :--- | :--- | :--- | :--- |
| `id` | String | 例: `abc340_c` (プライマリキー) | 一意の識別子 |
| `title` | String | 問題タイトル | UI表示用 |
| `url` | String | 問題ページのURL | UI表示用 |
| `difficulty` | Integer | 難易度（灰、茶など） | フィルタリング用 |
| `raw_text` | String | 生の問題文・解説テキスト | RAG生成時のLLM入力用 |
| `keywords` | String | Geminiが抽出したアルゴリズム名や要約 | **ベクトル化の元データ** |
| `embedding`| Vector | `keywords`を変換した数値配列 | 類似度検索用 |

## 3. 開発ロードマップ

### フェーズ0: 実験と検証 (PoC - Proof of Concept)
まずは1問（例: 直近のABCのC問題）をターゲットに、データパイプラインが通るか検証する。
- [ ] **APIテスト:** AtCoder Problems APIを叩き、JSONの構造を確認する。
- [ ] **スクレイピング検証:** ターゲット問題の問題文と、公式解説（テキストベース）のHTMLを取得し、BeautifulSoupでテキストを抽出する。※外部PDFや動画は除外。
- [ ] **抽出プロンプトの調整:** 取得した生テキストをGemini 2.5 Flash APIに投げ、「アルゴリズム名、データ構造、問題の要約」を抽出・JSON化させるプロンプトを確定する。
- [ ] **ベクトル化テスト:** 抽出したテキストをEmbedding APIに投げ、ベクトル（数値配列）が返ってくることを確認する。

### フェーズ1: データベース構築
フェーズ0で作った処理を、ローカルのDBに繋ぎ込む。
- [ ] ChromaDBのローカルクライアントを初期化し、保存先ディレクトリを設定する。
- [ ] フェーズ0で作成した1問分のデータ（メタデータ＋ベクトルデータ）をDBに `add`（Insert）する処理を実装する。
- [ ] DBからデータを `get` して、正しく保存されているか確認する。

### フェーズ2: 推論ワークフロー構築（精度検証）
実際に検索が機能するかをテストする基盤を作る。
- [ ] 検索クエリ（例：「木構造 最小公通祖先」）を受け取るPython関数を作成。
- [ ] クエリをEmbedding APIでベクトル化し、ChromaDBに対して類似検索（`query`）を実行する。
- [ ] **精度の精査:** 過去に解いた問題の特徴を入力し、意図した問題が上位にヒットするか検証。ヒットしない場合はフェーズ0の「抽出プロンプト」を改善する。

### フェーズ3: 更新ワークフロー構築（バッチ処理）
全件取得および定期更新の自動化スクリプトを作成する。
- [ ] AtCoder Problems APIから全問題リストを取得し、ローカルDBのID群と「差分」を取るロジックを実装。
- [ ] 未取得のIDに対して、フェーズ0の処理（スクレイピング → 抽出 → ベクトル化 → DB保存）をループで実行する。
- [ ] **【重要】マナー遵守:** ループ内には必ず `time.sleep(2)` 等を入れ、サーバーに負荷をかけない設計にする。
- [ ] **エラーハンドリング:** 途中でネットワークエラー等が起きても、次回再開できるように実装する（処理完了IDのログ出力など）。

### フェーズ4: フロントエンド構築 (UI)
完成した検索システムをブラウザから使いやすくする。
- [ ] Streamlitを用いて、検索窓と検索ボタンを配置したWeb UIを作成。
- [ ] 検索結果として、類似度が高い順に「問題タイトル、URL、Difficulty、抽出されたキーワード」をカード形式で表示させる。
- [ ] （オプション）検索結果の `raw_text` とユーザーの質問をLLMに投げ、対話形式でヒントをもらえるチャットUIを追加する。
