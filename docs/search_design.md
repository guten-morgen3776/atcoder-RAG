# AtCoder-RAG 類題検索機能 設計仕様書

## 1. 目的と概要

**目的:** 解いた問題から得た知見を一般化・言語化するため、既存のRAGシステムを拡張し、指定した問題とベクトル空間上で距離が近い「類題」を簡単に検索できる機能を実装する。

**方式:**
ChromaDBに既に保存されている各問題の埋め込み表現（Embedding）を活用する。事前の全ペア距離計算（$O(N^2)$）は行わず、検索実行時に基準問題のベクトルを取得して類似検索を行う（オンデマンドベクトル検索方式）。対象問題数は約1500問であるため、本方式で十分高速（数ミリ秒〜数十ミリ秒）に機能する。

## 2. UI/UX 設計 (`app.py` の改修)

既存の「キーワード検索」機能と共存させるため、Streamlitのタブ機能（`st.tabs`）を用いて画面を分割する。

### 2.1 共通ヘッダー
* **タイトル:** AtCoder-RAG
* **DB収録ステータス表示:**
  未収録問題の検索によるエラーを防ぐため、現在の収録問題数と対象範囲を表示する。
  *(表示例: `現在の収録データ: ABC126 〜 最新 (計1485問)`)*

### 2.2 タブ構成
* **タブ1: 「キーワードから検索」**（既存のUI）
* **タブ2: 「問題番号から類題検索」**（新規追加）

### 2.3 「類題検索」タブの入力フォーム設計
表記揺れをなくし、内部で正確な `problem_id` (例: `abc126_c`) を生成できるよう、UIコンポーネントを分割する。

1. **コンテスト種別 (`st.selectbox`):** `["ABC", "ARC", "AGC"]` (デフォルト: "ABC")
2. **コンテスト番号 (`st.number_input`):** 数字のみ許容（例: `126`）
3. **問題/大問選択 (`st.radio` - 水平配置):** `["A", "B", "C", "D", "E", "F", "G", "Ex"]`
4. **検索オプション:** * 難易度（Difficulty）フィルタ (`st.slider`)
   * 取得件数 Top K (`st.number_input`)
5. **実行ボタン:** `st.button("この問題の類題を探す")`

## 3. バックエンド処理フロー



「この問題の類題を探す」ボタン押下後の内部処理フロー。

1. **IDの正規化・結合:**
   入力値（例: "ABC", 126, "C"）から、一意のID文字列（例: `abc126_c`）を生成。
2. **存在チェック:**
   生成したIDがChromaDBに存在するか確認。存在しない場合は警告（`st.warning` 等）を出し、処理を中断する。
3. **基準ベクトルの取得:**
   ChromaDBに対し `collection.get(ids=["abc126_c"], include=["embeddings"])` を実行し、基準問題のベクトルデータを取り出す。
4. **類似ベクトル検索:**
   取得したベクトルをクエリとして、`collection.query(query_embeddings=[...], n_results=TopK + 1, where={難易度フィルタ})` を実行。
5. **自己除外処理:**
   検索結果の最上位（距離スコアが最小、自身とのマッチ）を除外する。
6. **結果の整形と表示:**
   取得した類似問題のメタデータ（タイトル、URL、難易度、アルゴリズム、キーワード）と、類似度スコア（距離）を描画する。
   * **UX向上施策:** 基準問題と検索結果の「アルゴリズム」「キーワード」を比較し、共通タグをハイライト表示する。
